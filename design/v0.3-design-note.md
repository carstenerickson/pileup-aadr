# v0.3 Design Note — Fan-out + Python Hot-Path Performance

**Status:** Pre-design (conceptual). Intended input to LLD session.
**Scope:** Four items grouped into one release; all touch the extract pipeline.

---

## Why these four items belong together

v0.3 has one large structural change (region-parallel Stage 3 fan-out) and three
Python hot-path improvements (iterrows replacement, concurrent swap_lookup build,
flat-dict rejoin lookup). The three hot-path items are small in isolation but
interact: items B and C share a data structure, and item A eliminates redundant
passes that overlap with fan-out's shard assembly logic. Designing them together
avoids a double-refactor.

---

## A. Region-parallel Stage 3 fan-out

### Current behavior

Stage 3 runs a single `samtools mpileup | pileupCaller` process across the entire
BED file. Both tools are single-threaded by nature (`samtools mpileup` has no `-@`
flag through v1.23.1; pileupCaller is Haskell with no parallelism). On a 33×
whole-genome BAM against the 1240k panel, Stage 3 takes 30–40 minutes.

The `--threads` CLI flag exists for historical reasons (it was mistakenly applied
to samtools in v0.1.0–v0.1.1, triggering issue #2) and currently emits a warning
that it is a no-op.

### Proposed change

Shard the sites BED into N regions, run N parallel `samtools mpileup | pileupCaller`
processes, merge the shard outputs, then proceed to Stage 4's rejoin as today.
Parallelism is bounded by `--threads` (repurposed) and hardware concurrency.

### Design decisions the LLD must resolve

**1. Shard definition**

Two candidate strategies:

- *Chromosome-split*: one shard per canonical chromosome (up to 25 for autosomes +
  sex chroms). Natural alignment to how bioinformatics tools process data; easy to
  reason about and test; poor load balance on human WGS (chr1 has ~5× the sites of
  chrY). With 4 threads, users get 4 active shards from 25 available.

- *N equal-weight BED chunks*: split the sorted BED into N roughly equal-site-count
  chunks, crossing chromosome boundaries freely. Better CPU utilization; harder to
  reason about (a chunk may span chr7..chr9); pileupCaller requires contiguous BAM
  regions so intra-chromosome splits must align to position, not line boundaries.

Decision needed: which strategy, or a two-level hybrid (chunk within chromosome,
N chunks per chromosome).

**2. Merge strategy for pileupCaller shard outputs**

Each shard produces a pileupCaller EIGENSTRAT triplet (.geno + .snp + .ind) in
the shard's region order. The final merged EIGENSTRAT triplet must preserve the
AADR row order that Stage 4 depends on (rsID-join requires AADR ordering, not
shard-output ordering).

Two candidate strategies:

- *Pre-Stage-4 merge*: concatenate all shard .geno and .snp files (in AADR row
  order) before Stage 4, so Stage 4's rejoin interface is unchanged. Costs one
  additional sequential pass over ~3 MB of shard outputs. Stage 4 is unmodified.

- *Per-shard Stage 4*: run the rejoin loop per shard, writing directly to the
  output prefix, merging during write. Eliminates the intermediate concat step;
  requires Stage 4's rejoin function to accept a list of shard prefixes and merge
  during write. More complex; rewrites the Stage 4 interface.

Decision needed: which strategy. The pre-Stage-4 merge is lower risk; the
per-shard approach couples the fan-out design to the Stage 4 redesign in item C.

**3. Site ordering invariant**

The current pipeline produces an EIGENSTRAT triplet where row N in .geno
corresponds to row N in .snp, and the .snp rows are in AADR's original order
(sorted by chrom_int then pos_bp within the AADR .snp file). This ordering is
relied on by downstream consumers (pgen-samplebind, AT2 sample-bind). The merge
step must preserve it. The LLD should specify exactly how shard ordering is tracked
(e.g., a shard manifest mapping shard index → AADR row range) to prevent silent
ordering bugs.

**4. `--threads` semantic change**

`--threads N` currently means "number of threads for samtools" (a no-op, with
warning). For fan-out it becomes "number of parallel Stage 3 shards." This is a
behavior change. The warning text changes. The LLD should address whether `N=1`
is a valid no-fan-out path (it should be, for single-core CI and debug) and what
the default is (1? auto-detect from `os.cpu_count()`?).

**5. Seed derivation per shard**

pileupCaller's `--seed` controls the random allele sampling; the master `--seed`
CLI argument must produce deterministic per-shard seeds so that a multi-shard run
is reproducible given the same inputs and `--threads`. The LLD must specify the
derivation (e.g., `shard_seed = master_seed + shard_index`) and whether this is
documented in the JSON report's `config` block.

**6. Error handling**

If one shard fails (pileupCaller non-zero exit, FASTA mismatch, corrupt BAM
region), remaining shards should be cancelled before raising. The existing
`ToolSubprocessError` taxonomy applies per-shard; the LLD should specify how the
first failure is surfaced (first-exception-wins, all-errors-aggregated, etc.) and
whether partial shard outputs are cleaned up on failure.

**7. Coverage gate aggregation**

Stage 4 currently computes `per_chrom_call_count` as a running dict during the
rejoin loop. With N shards, N `per_chrom_call_count` dicts must be merged before
the coverage gate runs. The merge is a simple key-wise sum; the LLD should confirm
it happens before `_evaluate_coverage_gate` and is represented correctly in the
JSON report.

**8. Progress reporting**

The current `log.info("Stage 3 complete: X/Y sites …")` fires once. With N
parallel shards, the log should report per-shard completions (DEBUG level, per
shard) and an aggregate summary (INFO level, once all shards complete). The LLD
should define the log schema for both.

## B. Replace `iterrows()` in hot-path iteration

### Current behavior

`build_sites_vcf` and `_write_aadr_native_snp_and_bed` both iterate the 1.2M-row
AADR DataFrame using `aadr_df.iterrows()`. Each call materializes a full
`pandas.Series` per row — 1.2M Series objects allocated and garbage-collected per
invocation. Measured characteristically 5–15× slower than `itertuples()` on
DataFrames of this size.

`build_sites_vcf` also collects all filtered rows into a `rows_to_write` list
(~1.1M tuples) and sorts them before writing. Whether this sort is necessary
depends on whether `parse_aadr_snp` guarantees a (chrom, pos)-sorted output —
a fact the LLD should verify against `format_detect.py`.

### Proposed change

Replace `iterrows()` with `df.itertuples()` (or vectorized boolean mask + array
walk) in both functions. Eliminate the `rows_to_write` sort if `parse_aadr_snp`
guarantees sorted output; otherwise document the guarantee requirement explicitly.

### Design decisions the LLD must resolve

**1. Sort elimination**

Does `parse_aadr_snp` return a DataFrame sorted by (chrom_int, pos_bp)? If yes,
the sort in `build_sites_vcf` is redundant and should be removed. If no, the sort
should be moved to `parse_aadr_snp` (paid once, at parse time) rather than repeated
in each call site.

**2. Vectorized vs itertuples**

For `build_sites_vcf`'s filter step (palindrome filter, non-SNP filter), a
vectorized pandas boolean mask + `.to_records()` walk is faster than `itertuples()`
and may allow eliminating `rows_to_write` entirely (streaming write after mask
application). The LLD should pick one approach and apply it consistently.

## C. Concurrent `_build_swap_lookup` during Stage 3

### Current behavior

`_build_swap_lookup(lifted_vcf_path)` is called at the start of `rejoin_aadr_frame`
(Stage 4), after Stage 3 completes. It walks the entire lifted VCF through pysam
(~1.1M records) to build `{rsid: bool}`. The lifted VCF exists at the end of
Stage 1 and sits idle for the full 30–40 minute Stage 3 runtime — the lookup
dict could be built during that window at zero additional wallclock cost.

This item applies only to the full lift path. The no-lift fast path has no lifted
VCF and no swap_lookup.

### Proposed change

Start a background worker immediately after Stage 1 completes. The worker calls
`_build_swap_lookup` on the lifted VCF. Stage 4 waits for (joins) the worker
before starting the rejoin loop.

### Design decisions the LLD must resolve

**1. Threading model and exception propagation**

A bare `threading.Thread` with a result stored in a shared list is error-prone:
exceptions in the thread do not propagate to the main thread on `.join()`. The
recommended model is `concurrent.futures.ThreadPoolExecutor` submitting a single
`Future`; `future.result()` at Stage 4 entry re-raises any exception cleanly in
the main thread with full traceback. The LLD should specify this pattern, including
the executor's context manager scope within the orchestrator.

**2. pysam thread safety**

pysam/htslib is not thread-safe for shared file handles, but opening a new
`pysam.VariantFile` in the worker thread (as `_build_swap_lookup` already does)
is safe. The LLD should confirm no htslib objects are shared between the worker
and the main thread.

**3. Interaction with item D (flat-dict rejoin)**

See §Cross-cutting below. The swap_lookup result from this worker feeds directly
into the merged lookup dict in item D. The LLD should specify the handoff point:
the swap_lookup Future is resolved once, then merged into the aadr_lookup dict,
before the Stage 4 loop begins.

## D. Flat-dict lookup in Stage 4 rejoin loop

### Current behavior

`rejoin_aadr_frame` calls `aadr_df.loc[rsid]` once per variant (1.1M calls). Each
call dispatches through pandas indexing machinery and returns a `pandas.Series`.
The Series is then unpacked field by field (`row["chrom_int"]`, `row["ref"]`, etc.).
Additionally, the swap_lookup dict is queried separately: `swap_lookup.get(rsid, False)`.
This results in two Python object lookups per variant in the hot path.

### Proposed change

Before the rejoin loop, build a single compact Python dict:
`{rsid: (chrom_int, gen_morgans, pos_bp, ref, alt, is_swapped)}`. The loop then
does one dict lookup per variant and destructures a tuple — no pandas overhead,
no second dict lookup, fewer Python object allocations per iteration.

The AADR DataFrame is still needed for `classify_aadr_chrom_set` and the coverage
gate denominator computation; it should be kept alive until those calls complete.
After the merged dict is built, the DataFrame may be released explicitly.

### Design decisions the LLD must resolve

**1. Dict build strategy**

The merged dict is built from two sources: the AADR DataFrame (rows) and the
swap_lookup (bool flags, available from item C's Future). The LLD should specify
the build order (resolve the Future first, then build the merged dict in one pass
over the DataFrame using `itertuples()`), and the tuple field layout (fixed order,
documented).

**2. Memory release**

After building the merged dict, the aadr_df and swap_lookup dict (now redundant)
may be explicitly deleted. The LLD should specify whether `del` calls are used or
whether the orchestrator's scope naturally handles release, and verify no other
call site retains a reference to aadr_df after Stage 4 begins.

## Cross-cutting concerns

**Items B, C, D share `aadr_df`'s lifecycle.** The merged dict in item D is built
from aadr_df + item C's swap_lookup Future. The sort-elimination decision in item
B may change the column layout of aadr_df rows. These three items should be
designed together and landed in a single coordinated commit, not three separate
PRs that each modify the hot path.

**Fan-out and items C/D interact at the merge boundary.** If the fan-out uses the
pre-Stage-4 merge strategy (concatenate shard outputs before rejoin), items C and D
are structurally unchanged — Stage 4 still sees a single stream. If the fan-out
uses the per-shard Stage 4 strategy, item D's dict must be built once and shared
across N per-shard rejoin calls (read-only after build, so threading is safe). The
LLD should pick the fan-out merge strategy first, then specify how items C and D
fit against it.

**No-lift fast path is untouched by items C and D.** The fast-path finalizer
(`_no_lift_fast_path_finalize`) does not use a swap_lookup and already copies files
rather than joining. Item B's iterrows fix applies to `_write_aadr_native_snp_and_bed`,
which is the fast path's analog of `build_sites_vcf`. The LLD should confirm both
call sites are updated consistently.

## What the LLD should deliver

For each item:

- Data structure specifications (field layout, types, dict key contracts)
- Control flow diagram or sequence showing where concurrent work starts/joins
- Module-level interface changes (new parameters, changed return types)
- Invariants the implementation must preserve
- Test strategy: which new unit tests, which existing tests must continue to pass,
  whether any integration test baselines change (Stage 3 counter values, JSON
  report schema, EIGENSTRAT output byte counts)

For the fan-out specifically:

- Shard definition algorithm (pseudocode)
- Shard manifest schema
- Merge algorithm with site ordering proof sketch
- `--threads` documentation update (CLI help string, README)
- JSON report schema changes (per-shard counters vs aggregate)
